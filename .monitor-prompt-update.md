**Doc template — follow this exactly:**
```
# [Feature Name]

**Category:** [category name]
**Applies To:** [Standalone / Embedded / Both]
**Source:** [MS Learn URL]

## What it does
[Direct statement of what the feature is. 1–3 sentences. No "This feature provides..." or "enables organisations to..." — just state what it is.]

## Key facts
- [bullet list of limits, defaults, prerequisites, non-obvious behaviour]

## When to use / skip
[Blunt judgment call. "If you're doing X, you need this. Skip it if Y." Not a decision tree.]

## Configuration decisions
- [bullet list of decisions an implementer has to make]

## Gotchas
- [Notes from experience, not warning labels. Drop "It is important to note..." — just say the thing.]

---

*[Review trigger in natural language — e.g. "Worth revisiting after the next release wave"]*
```

**Writing style rules — apply to every doc:**
- Direct, experienced, slightly dry, British English
- No: leverage, utilise, robust, seamless, comprehensive, facilitate, ensure → use "use", "solid", "make sure"
- No: "it is important to note", "Note that", "implementers should"
- Use "you'll", "you're" — peer-to-peer tone
- Gotchas: sound like notes from someone who's been caught out, not a warning label
- When to use/skip: blunt judgment call, not a flowchart
